Description de l'Application

Nom
assistant RH
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'assistant RH est une application web interne de niveau critique medium, conçue pour répondre aux questions des employés sur les congés, politiques internes et démarches administratives via un chatbot intégré. Architecturée en trois tiers avec une approche microservices, elle repose sur un frontend React communiquant en REST avec un backend Spring Boot, tandis que chaque microservice dispose de sa propre base de données PostgreSQL on-premise, incluant une instance dédiée pgvector pour le stockage des embeddings locaux. L'application exploite un modèle LLM externe Azure OpenAI en lecture seule, consommé via API et enrichi par une architecture RAG s'appuyant sur les documents internes indexés dans pgvector, sans possibilité d'appeler des outils externes ou de déclencher des actions automatisées. Les microservices communiquent entre eux via gRPC sans broker intermédiaire, et aucun flux sortant (emails, uploads) ou centralisation des logs n'est implémenté. Les données sensibles, exclusivement des documents internes, sont isolées dans la zone on-premise, tandis que l'interaction utilisateur se limite à un navigateur web sans accès direct aux composants backend ou LLM. Les principaux risques à modéliser concernent l'exposition des données sensibles via le chatbot, les échanges gRPC internes non sécurisés, l'intégrité des embeddings locaux, et la dépendance à l'API externe Azure OpenAI pour les réponses en temps réel.

Direct prompt injection Un attaquant modifie les instructions du prompt pour influencer les réponses du LLM, le faisant générer des informations biaisées ou inappropriées.
Indirect prompt injection Un attaquant insère des instructions cachées dans un document interne (ex: une politique RH) qui est ensuite indexé dans pgvector. Lors d'une requête RAG, le LLM récupère ce document et interprète les instructions malveillantes.
Indirect prompt injection Un attaquant injecte du contenu malveillant dans un document stocké dans la base de connaissances (pgvector) utilisée par le modèle RAG, provoquant des réponses inattendues ou inappropriées du chatbot.
Indirect prompt injection Un attaquant compromet une source de données internes et y insère des phrases déguisées en instructions pour le LLM, qui sont ensuite récupérées par le mécanisme RAG.
Long-term Memory Poisoning Un attaquant dépose un document malveillant dans une source indexée par l'assistant RH. Lors d’une recherche de contexte, le LLM récupère ce document et l’intègre à son traitement comme une donnée de référence falsifiée.
Long-term Memory Poisoning L’attaquant exploite un défaut d’authentification ou d’autorisation sur l’API de la base pgvector et modifie directement les vecteurs, les métadonnées et les entrées de mémoire à la source.
Long-term Memory Poisoning L’attaquant injecte un contenu malveillant qui n’est pas écrit immédiatement en mémoire, mais qui est intégré plus tard lors d’un processus de consolidation, de re-embedding ou de summarization périodique des documents internes.
Model denial of service Un attaquant envoie un grand nombre de requêtes simultanées à l'API Azure OpenAI via l'assistant RH pour remplir la file d’attente du modèle et empêcher le traitement des requêtes légitimes.
Model denial of service Un attaquant envoie des entrées (prompts ou documents RAG) dépassant la taille maximale du contexte pour provoquer des erreurs ou une dégradation du traitement par le LLM ou le pipeline RAG.
Model denial of service Un attaquant soumet des entrées spécialement conçues pour maximiser le temps de calcul du LLM externe ou des processus de recherche/embedding dans pgvector, épuisant les ressources et augmentant les coûts.
ML model evasion Un attaquant fournit des données (prompts) en dehors du domaine d’entraînement ou du comportement attendu du LLM pour provoquer des erreurs de classification ou contourner les filtres de sécurité.
ML model evasion Un attaquant manipule une entrée du chatbot pour qu’elle soit classée comme une catégorie spécifique choisie, afin d'obtenir une réponse ou une action spécifique et non autorisée du LLM.
ML model evasion Un attaquant exploite les scores ou réponses du LLM pour affiner ses entrées (prompts) et améliorer l’évasion des mécanismes de détection de sécurité.
Model stealing Un attaquant envoie un grand nombre de requêtes à l'API Azure OpenAI pour apprendre progressivement le comportement du LLM et reconstruire un modèle similaire.
Model stealing Un attaquant analyse les réponses du LLM pour déduire ses hyperparamètres internes ou son architecture.
Model stealing Un attaquant accède directement à la base de données pgvector pour copier les embeddings des documents internes, volant ainsi la 'connaissance' de l'assistant RH.
Model Tampering Un attaquant modifie directement les embeddings du modèle ou les documents sources stockés dans pgvector pour altérer le comportement du RAG.
Model Tampering Un attaquant injecte une modification dans les documents internes indexés pour que l'assistant RH réagisse de manière spécifique à un trigger caché via la RAG.
Model Tampering Un attaquant exploite une faille pour forcer le retour vers une ancienne version vulnérable des embeddings ou des documents sources, réintroduisant un biais ou une vulnérabilité.
Short-term Memory Poisoning L'attaquant injecte du contenu malveillant dans les segments de la mémoire de travail du chatbot, comme le 'scratchpad' ou les résultats temporaires de recherche RAG.
Short-term Memory Poisoning L'attaquant insère des instructions persistantes ou des faits falsifiés dans une conversation longue pour forcer leur inclusion lors de la phase de résumé (compression) du contexte du chatbot.
Short-term Memory Poisoning Un attaquant manipule les entrées successives du chatbot pour progressivement introduire un biais dans sa compréhension du contexte et de l'intention de l'utilisateur.
