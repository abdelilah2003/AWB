Description de l'Application

Nom
assistant RH
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'assistant RH est une application web interne de niveau critique moyen, conçue pour répondre aux questions des employés sur les politiques RH et automatiser les réponses administratives via un chatbot. Architecturée en trois tiers avec une approche microservices, elle s'appuie sur un frontend React communiquant en REST avec un backend Spring Boot, tandis que chaque microservice dispose de sa propre base de données PostgreSQL on-premise, incluant une instance dédiée pgvector pour le stockage vectoriel des embeddings locaux. L'intégration d'Azure OpenAI en lecture seule, consommé via API externe, permet d'enrichir les réponses grâce à une architecture RAG interrogeant les documents internes indexés dans pgvector, sans possibilité d'appeler des outils externes ou d'effectuer des actions modificatrices. Les microservices communiquent entre eux via gRPC sans broker intermédiaire, et l'application ne gère ni uploads, ni emails, ni tâches planifiées, avec des logs non centralisés. Les données sensibles (documents internes) restent strictement confinées dans l'infrastructure locale, tandis que le flux principal relie l'utilisateur au frontend, puis aux services backend qui orchestrent les requêtes vers le LLM et les bases relationnelles/vectorielles, sans exposition directe aux API externes autres qu'Azure OpenAI. Cette structure minimise les surfaces d'attaque mais impose une vigilance accrue sur les échanges gRPC internes, l'accès aux données sensibles et l'isolation des composants LLM/RAG.

Menaces Retenues du Catalogue

Menaces issues du catalogue et retenues selon le contexte technique et fonctionnel de l application.
Menace Description
SQL injection L'assistant RH utilise des bases de données PostgreSQL on-premise pour chaque microservice, y compris une instance pgvector pour le stockage vectoriel. Cette menace consiste à injecter des requêtes SQL malveillantes via les entrées utilisateur du chatbot ou d'autres formulaires de l'application web, visant à manipuler les requêtes de la base de données relationnelle ou vectorielle. Un attaquant pourrait ainsi extraire des informations sensibles, modifier des données ou contourner l'authentification.
Broken Authentication L'application assistant RH est une application web interne, ce qui signifie que les utilisateurs doivent s'authentifier. Des failles dans les mécanismes d'authentification de l'application web ou des API internes (par exemple, des politiques de mot de passe faibles, l'absence de re-authentification, des mécanismes de récupération de compte vulnérables) pourraient permettre à un attaquant de prendre le contrôle de comptes d'employés.
Cross-Site Scripting ( XSS ) L'application assistant RH est une application web React, où les entrées utilisateur sont affichées dans l'interface du chatbot ou d'autres composants. Si ces entrées ne sont pas correctement validées et échappées côté serveur et/ou client, un attaquant pourrait injecter du code JavaScript malveillant qui serait exécuté dans le navigateur d'autres utilisateurs internes.
Cross-Site Request Forgery En tant qu'application web interne, l'assistant RH est susceptible à la CSRF. Cette menace consiste à forcer un utilisateur authentifié à exécuter une action non désirée sur l'application, à son insu. Cela pourrait se produire si l'application ne vérifie pas l'origine des requêtes ou n'utilise pas de jetons anti-CSRF.
Directory Traversal Bien que l'application ne gère pas d'uploads, elle interroge des documents internes stockés dans l'infrastructure locale (indexés dans pgvector). Si la logique de récupération ou de lecture de ces documents utilise des chemins de fichiers basés sur des entrées contrôlées par l'utilisateur sans validation adéquate, un attaquant pourrait manipuler ces chemins pour accéder à des fichiers ou répertoires en dehors du répertoire autorisé.
Deserialization injection L'architecture microservices de l'assistant RH utilise gRPC pour la communication inter-services, et le backend est en Spring Boot (Java). Si des données non fiables sont désérialisées sans validation adéquate (par exemple, via des objets Java sérialisés ou des payloads JSON/XML complexes), un attaquant pourrait injecter des objets malveillants ou exécuter du code à distance.
Distributed Denial of Service L'application assistant RH est une application web et un chatbot LLM/RAG, ce qui la rend vulnérable aux attaques par déni de service distribué. Un attaquant pourrait saturer les ressources du frontend, du backend, des bases de données ou du LLM externe (Azure OpenAI) en envoyant un grand volume de trafic ou de requêtes coûteuses.
Direct prompt injection L'assistant RH est un chatbot basé sur un LLM où les employés peuvent poser des questions. Cette menace consiste à injecter des instructions malveillantes directement dans le prompt du chatbot afin de manipuler le comportement du LLM, le faisant potentiellement ignorer ses instructions système ou divulguer des informations sensibles.
Indirect prompt injection L'assistant RH utilise une architecture RAG qui interroge des documents internes indexés dans pgvector. Cette menace consiste à injecter des instructions malveillantes non pas directement dans le prompt de l'utilisateur, mais dans ces documents internes eux-mêmes (par exemple, par un employé malveillant ou une compromission antérieure). Le LLM pourrait ensuite lire et exécuter ces instructions cachées indirectement.
Information Disclosure L'application assistant RH manipule des données RH sensibles. Des erreurs de configuration, des messages d'erreur détaillés, des logs non centralisés ou une mauvaise gestion des accès pourraient exposer ces informations à des utilisateurs non autorisés, même internes.
Jwt manipulation L'application web React communique en REST avec un backend Spring Boot, ce qui implique potentiellement l'utilisation de JSON Web Tokens (JWT) pour l'authentification et l'autorisation des utilisateurs internes. Des failles dans la gestion des JWT pourraient permettre à un attaquant de modifier les claims (par exemple, les rôles) ou de forger des tokens valides.
Broken Object Level Authorization Dans une architecture microservices où chaque service possède sa propre base de données PostgreSQL, il est crucial que chaque microservice applique un contrôle de propriété strict sur les ressources accédées. Si l'API d'un microservice (REST ou gRPC) ne vérifie pas que l'utilisateur demandeur est autorisé à accéder à un objet spécifique (par exemple, un document RH, un dossier employé), un attaquant pourrait manipuler les identifiants pour accéder ou modifier les objets d'autrui.
Broken Object Property Level Authorization L'application assistant RH expose des APIs REST via un backend Spring Boot. Si ces APIs exposent des propriétés d'objets sensibles dans leurs réponses ou acceptent leur modification sans un contrôle granulaire au niveau du champ, un attaquant pourrait collecter des informations sensibles ou altérer des données critiques en injectant des propriétés non autorisées.
IDOR exploitation Dans une application microservices comme l'assistant RH, qui gère des données sensibles d'employés, l'exploitation d'IDOR (Insecure Direct Object Reference) est un risque élevé. Elle consiste à manipuler des identifiants d'objets (dans les URLs, paramètres de requête, corps de requêtes) pour accéder ou agir sur des ressources sans autorisation, en l'absence de vérification de propriété côté serveur.
Man-in-the-Middle Les communications au sein de l'application assistant RH incluent le frontend vers le backend (REST), les microservices entre eux (gRPC) et le backend vers Azure OpenAI (API externe). Une attaque Man-in-the-Middle (MITM) pourrait intercepter et éventuellement modifier ces communications si elles ne sont pas suffisamment chiffrées et authentifiées, ou si les certificats ne sont pas validés.
ML model evasion L'assistant RH utilise un LLM (Azure OpenAI) et une base d'embeddings locaux (pgvector) via RAG. Un attaquant pourrait manipuler les entrées du chatbot ou les données des documents RAG afin de contourner les prédictions ou décisions attendues du modèle lors de l'inférence. Cela peut conduire à des réponses incorrectes ou non conformes.
Model denial of service L'assistant RH dépend fortement du LLM externe Azure OpenAI et des bases de données pgvector pour le RAG. Un attaquant pourrait exploiter les limites computationnelles ou opérationnelles de ces composants en générant une charge excessive ou anormale, afin de dégrader leurs performances, épuiser les ressources et provoquer l'indisponibilité du service de chatbot.
Model stealing L'application assistant RH utilise des modèles d'embedding locaux stockés dans pgvector et consomme un LLM externe. Bien que le LLM lui-même soit externe, un attaquant pourrait tenter d'accéder, de copier ou de reconstruire les modèles d'embedding locaux (pgvector) ou de déduire le comportement du LLM externe par des requêtes API répétées, afin d'exploiter la propriété intellectuelle ou de faciliter d'autres attaques.
AI supply chain tampering L'assistant RH utilise une architecture RAG avec des embeddings locaux (pgvector) et consomme Azure OpenAI. Cette application repose sur des composants d'IA (modèles d'embedding, pipelines RAG, librairies). Une compromission de la chaîne d'approvisionnement (par exemple, en introduisant un composant malveillant dans les librairies utilisées pour les embeddings ou le RAG) pourrait introduire un comportement hostile dans le flux de confiance.
Model Tampering L'assistant RH utilise une base d'embeddings locaux (pgvector) qui peut être considérée comme un 'modèle' dans ce contexte, et consomme Azure OpenAI. Cette menace consiste à compromettre l’intégrité des embeddings locaux, leurs artefacts de génération, ou les processus de mise à jour, afin d’introduire un biais ou un comportement malveillant persistant dans le système RAG. Cela pourrait également cibler la configuration d'interaction avec le LLM externe.
Password Reset takeover Si l'application assistant RH permet aux utilisateurs internes de réinitialiser leurs mots de passe, des failles dans ce mécanisme pourraient permettre à un attaquant de prendre le contrôle du compte d'un employé sans disposer de ses identifiants initiaux.
Race Condition L'architecture microservices de l'assistant RH avec des communications gRPC et des bases de données distribuées (chaque microservice a sa propre DB) est susceptible aux 'race conditions'. Cela se produit lorsque plusieurs requêtes tentent de modifier une ressource partagée en même temps, entraînant un état incohérent ou des actions non autorisées si les mécanismes de synchronisation sont absents ou défectueux.
Server-Side Request Forgery ( SSRF ) L'application assistant RH consomme une API externe (Azure OpenAI). Si les requêtes envoyées à cette API externe sont construites dynamiquement à partir d'entrées utilisateur sans validation stricte, un attaquant pourrait manipuler ces entrées pour forcer le backend à envoyer des requêtes vers des ressources internes (autres microservices, bases de données locales) ou vers d'autres services externes non autorisés.
Server-Side Template Injection Bien que le frontend soit en React, l'application web peut générer du contenu HTML côté serveur pour certaines parties (par exemple, des pages d'erreur personnalisées, des rapports simples si cela évolue). Si des entrées utilisateur sont directement injectées dans des templates côté serveur sans échappement ni validation, un attaquant pourrait injecter du code malveillant qui sera exécuté par le moteur de template, menant à l'exécution de code ou à la divulgation d'informations.
Session Fixation L'application assistant RH gère des sessions pour ses utilisateurs internes. Cette menace consiste à forcer un utilisateur à utiliser un identifiant de session connu par l’attaquant avant l'authentification. Une fois l'utilisateur connecté avec cet ID fixé, l'attaquant peut utiliser l'ID de session pré-assigné pour prendre le contrôle de la session authentifiée.
Short-term Memory Poisoning L'application assistant RH utilise une architecture RAG avec des embeddings locaux stockés dans pgvector et un LLM. La 'short-term memory' fait référence au contexte de la conversation et aux informations temporaires utilisées par le LLM pour générer des réponses. Cette menace consiste à injecter ou altérer des informations malveillantes dans cette mémoire temporaire afin d’influencer le raisonnement ou les décisions du chatbot pendant la session en cours.
Long-term Memory Poisoning L'assistant RH utilise pgvector pour le stockage vectoriel des embeddings de documents internes, constituant sa 'long-term memory' pour le RAG. Cette menace consiste à injecter ou altérer des informations malveillantes dans cette mémoire persistante (les embeddings ou les documents source) afin d’influencer durablement le comportement du chatbot et ses réponses, même au-delà d'une seule session.
User Impersonation L'assistant RH est utilisé par des employés internes. Cette menace consiste à usurper l’identité d’un utilisateur légitime afin d’obtenir un accès non autorisé à l'application et de faire exécuter des actions par le chatbot au nom de cet utilisateur. Cela inclut le vol de credentials ou de sessions.
WebSocket Exploitation Le chatbot de l'assistant RH utilise probablement des communications en temps réel, ce qui pourrait impliquer des WebSockets. Des failles dans l'implémentation des WebSockets (par exemple, validation des messages, contrôles d'origine, gestion des sessions) pourraient permettre à un attaquant d'interagir avec le serveur de manière non autorisée, d'injecter du code ou de mener des attaques par déni de service.

Server-Side Template Injection Un attaquant injecte une expression dans un champ de saisie (par exemple, dans un profil utilisateur ou une requête de recherche) qui est ensuite rendue par un template côté serveur, exécutant ainsi du code sur le serveur.
Server-Side Template Injection Un attaquant exploite le moteur de template pour accéder à des variables internes du serveur (configurations, clés API) ou des fichiers système.
Server-Side Template Injection Un attaquant injecte du code pour modifier le contenu HTML généré par le template, par exemple en falsifiant une réponse affichée à l'utilisateur ou en insérant des liens malveillants.
Session Fixation Un attaquant envoie un lien malveillant à la victime contenant un ID de session prédéfini. La victime clique, est authentifiée avec cet ID, et l'attaquant utilise ensuite cet ID pour se connecter en tant que victime.
Session Fixation Un attaquant exploite une vulnérabilité Cross-Site Scripting (XSS) ou une mauvaise configuration de l'application pour injecter un ID de session spécifique dans le navigateur de la victime avant son authentification.
Session Fixation L'attaquant impose un ID de session à la victime (via URL, cookie manipulé) avant son authentification, puis attend qu'elle se connecte pour réutiliser cet ID.
Short-term Memory Poisoning L'attaquant injecte du contenu malveillant dans les segments de la mémoire de travail de l'agent (chatbot), comme le 'scratchpad' ou les résultats intermédiaires du RAG, pour influencer les étapes de son raisonnement.
Short-term Memory Poisoning L'attaquant insère des instructions persistantes ou des faits falsifiés dans une conversation longue pour forcer leur inclusion lors de la phase de résumé (compression) du contexte, qui est ensuite utilisée par le LLM.
Short-term Memory Poisoning Un attaquant manipule les métadonnées ou les réponses intermédiaires générées par le pipeline RAG pour 'empoisonner' la mémoire à court terme du LLM avec des informations biaisées.
Long-term Memory Poisoning L’attaquant dépose ou modifie un document malveillant dans une source indexée par l’agent RAG. Lors d’une recherche de contexte, l’agent récupère ce document et l’intègre à son traitement comme une donnée de référence falsifiée.
Long-term Memory Poisoning L’attaquant exploite un défaut d’authentification ou d’autorisation sur l’API d'ingestion/gestion de pgvector. Il accède directement aux enregistrements stockés puis modifie les vecteurs ou les métadonnées à la source.
Long-term Memory Poisoning L’attaquant injecte un contenu malveillant qui n’est pas écrit immédiatement en mémoire, mais qui est intégré plus tard lors d’un processus de consolidation, de ré-indexation ou de 're-embedding' périodique dans pgvector.
User Impersonation L’attaquant obtient les identifiants ou facteurs d’authentification d’un utilisateur légitime (par exemple, via phishing ou vol de mots de passe), puis se connecte à l’assistant RH sous cette identité pour interagir avec le chatbot ou accéder aux ressources.
User Impersonation L’attaquant récupère un cookie de session ou un jeton de session associé à une session déjà authentifiée, puis le réutilise pour reprendre l’interaction avec l’agent sous l’identité de l’utilisateur légitime.
User Impersonation L’attaquant injecte dans le contexte de l’agent (par exemple, dans un prompt ou un message) une fausse assertion d’identité, de rôle ou d’autorité utilisateur, puis amène le chatbot à traiter cette déclaration comme une preuve valide sans vérification indépendante.
WebSocket Exploitation Un attaquant force le navigateur de la victime à établir une connexion WebSocket vers l'assistant RH en utilisant ses cookies de session (Cross-Site WebSocket Hijacking - CSWSH).
WebSocket Exploitation Un attaquant intercepte et modifie les messages WebSocket échangés entre le client et le serveur du chatbot pour injecter des commandes ou des requêtes malveilluses.
WebSocket Exploitation Un attaquant ouvre un grand nombre de connexions WebSocket persistantes pour épuiser les ressources du serveur, provoquant un déni de service pour les utilisateurs légitimes.

Mitigation à appliquer
Utiliser exclusivement des requêtes préparées pour toutes les interactions avec les bases de données PostgreSQL, y compris pgvector.
Valider et nettoyer strictement toutes les entrées utilisateur côté serveur, en contrôlant le type, le format, la taille et le contenu des paramètres.

Déployer un WAF avec des règles de détection et de blocage des injections SQL.
Chiffrer les données sensibles au repos dans les bases de données PostgreSQL.
Journaliser tous les événements de sécurité liés aux bases de données et centraliser les traces dans un SIEM pour la détection d'anomalies.
Appliquer un rate limiting strict par compte, IP et session sur les endpoints d’authentification.
Imposer un MFA robuste sur les flux de login, réinitialisation et récupération de compte.
Imposer une longueur minimale élevée et interdire les mots de passe à faible complexité, en les comparant à des corpus de secrets compromis connus.
Générer tous les tokens et identifiants de session avec un générateur cryptographique sécurisé (CSPRNG) et s'assurer d'une entropie suffisante.
Vérifier systématiquement la signature cryptographique avant toute acceptation d'un jeton et valider l’algorithme JWT côté serveur via une allowlist stricte.
Implémenter un mécanisme d’échappement systématique de toutes les données en sortie, adapté au contexte d'affichage (HTML, JavaScript, URL).
Valider et filtrer strictement toutes les entrées utilisateur côté serveur et côté client, en utilisant des listes blanches de caractères autorisés.
Implémenter une politique de sécurité de contenu (CSP) stricte pour restreindre l’exécution de scripts non autorisés et le chargement de ressources depuis des origines non fiables.
Déployer un WAF avec des règles de détection et de blocage des attaques XSS.
Activer les attributs de sécurité des cookies (HttpOnly, Secure, SameSite=Strict) pour protéger les identifiants de session.
Implémenter des jetons anti-CSRF cryptographiquement forts, uniques par requête (one-time token), à inclure dans toutes les requêtes d'état modifiant (POST, PUT, DELETE).
Utiliser des cookies de session avec l'attribut SameSite=Strict par défaut pour empêcher l'envoi de cookies lors de requêtes cross-site.
Vérifier l'en-tête `Origin` de toutes les requêtes pour s'assurer qu'elles proviennent d'un domaine approuvé.
Exiger une re-authentification forte (step-up authentication) pour toute action critique ou sensible.
Implémenter une validation stricte des chemins de fichiers basés sur une liste blanche de répertoires et de noms de fichiers autorisés.
Utiliser des identifiants indirects (UUID) pour les documents plutôt que des chemins de fichiers directement manipulables par l'utilisateur.
Stocker les fichiers sensibles (configurations, clés) en dehors des répertoires accessibles par l'application web et le service de recherche RAG.
Appliquer le principe du moindre privilège sur les accès aux fichiers pour l'application, en limitant les droits aux seuls fichiers nécessaires.
Désactiver l’indexation des répertoires sur le serveur web et configurer le serveur pour interdire l’accès aux fichiers sensibles (.env, config).
Éviter les mécanismes de désérialisation natifs non sécurisés et utiliser des formats de données sécurisés (par exemple, JSON ou Protobuf avec validation de schéma strict) pour les communications inter-services.
Implémenter une allowlist stricte des classes ou types de données autorisés pour la désérialisation.
Signer toutes les données sérialisées (HMAC ou signature numérique) et vérifier leur intégrité avant toute désérialisation.
Limiter la profondeur et la taille des objets désérialisés pour prévenir les attaques par déni de service.
Isoler les composants traitant des données sérialisées dans des environnements sécurisés et appliquer le principe du moindre privilège.
Déployer une protection anti-DDoS en amont du réseau et utiliser un CDN pour absorber le trafic.
Implémenter un rate limiting strict par IP, endpoint, utilisateur et clé API sur tous les points d'entrée (frontend, API backend).
Déployer un WAF avec protection anti-DDoS applicatif et détection des requêtes automatisées.
Implémenter un autoscaling dynamique des microservices backend pour gérer les pics de charge.
Définir des quotas d'usage (par exemple, nombre de tokens) par utilisateur ou session pour les interactions avec le LLM.
Implémenter des timeouts et circuit breakers entre les microservices pour éviter la propagation d'une défaillance.
Stocker le 'system prompt' uniquement côté backend et l'isoler des entrées utilisateur, empêchant toute modification ou divulgation par ces dernières.
Séparer clairement les instructions système et les données utilisateur dans le prompt, en appliquant une hiérarchie stricte.
Filtrer et nettoyer tous les prompts utilisateur avant envoi au LLM, en supprimant les balises, caractères spéciaux et patterns d'injection connus.
Déployer des 'guardrails' pour contrôler les entrées utilisateur et les réponses du modèle, bloquant les contenus interdits ou demandant des secrets.
Limiter le contexte conversationnel réutilisé en cas de prompt suspect et réinitialiser ou isoler le contexte après détection d’une tentative d’injection.
Valider et filtrer toutes les données externes (documents internes) avant leur ingestion et indexation dans la base pgvector, en scannant les contenus pour détecter du code malveillant ou des instructions cachées.
Sanitiser les documents en supprimant scripts, HTML actif et tout contenu dynamique avant l'indexation RAG.
Restreindre l’ingestion de données aux sources approuvées et de confiance, avec une traçabilité complète de la provenance des documents.
Implémenter un pipeline de sanitization robustes avant l'indexation dans pgvector.
Déployer des 'guardrails' de sortie pour filtrer toutes les réponses du LLM, empêchant la divulgation de données sensibles ou d'instructions système, et appliquant des politiques AI-DLP.
Classifier les données sensibles manipulées par l'application et appliquer des contrôles d'accès stricts en fonction de cette classification.
Masquer ou tronquer les données sensibles dans les interfaces utilisateur et les journaux en production. Désactiver les messages d’erreur détaillés en production pour n'afficher que des messages génériques.
Éviter toute exposition de secrets, clés, mots de passe et tokens, en les stockant dans un gestionnaire de secrets sécurisé (vault) et en les injectant dynamiquement.
Durcir les serveurs web et applicatifs selon les benchmarks de sécurité (CIS Benchmark) et supprimer les fichiers de debug, logs et backups accessibles publiquement.
Implémenter une solution de DLP (Data Loss Prevention) pour détecter et bloquer les fuites d’information, en particulier pour les données RH.
Vérifier systématiquement la signature cryptographique de chaque JWT avant toute utilisation et refuser tout token avec `alg=none`.
Restreindre les algorithmes de signature autorisés (par exemple, RS256, ES256 uniquement) et protéger, faire tourner et cloisonner les clés de signature dans un stockage sécurisé (KMS/Vault).
Utiliser des tokens d'accès courts (expiration courte) et des refresh tokens sécurisés, avec un mécanisme de détection de réutilisation.
Ne jamais faire confiance au contenu du payload sans validation côté serveur (par exemple, re-vérifier les rôles et permissions côté backend avant toute action sensible).
Transmettre les JWT uniquement via HTTPS (TLS obligatoire) et ne jamais les exposer dans les URLs ou logs.
Vérifier côté serveur que l'identifiant de ressource (document RH, dossier employé) appartient à l'utilisateur authentifié (via le jeton JWT ou la session), à chaque appel API.
Ne jamais extraire l'identité de l'appelant depuis la requête elle-même, mais toujours la lire depuis un jeton signé ou une session sécurisée.
Générer les identifiants de ressources en UUID v4 aléatoires pour empêcher l'énumération.
Implémenter un contrôle d’accès par rôles (RBAC) ou par attributs (ABAC) granulaire au niveau de chaque microservice pour limiter chaque utilisateur aux actions et ressources autorisées selon son rôle et ses droits.
Isoler les microservices par domaine de données et implémenter une autorisation basée sur des politiques (par exemple, OPA) au niveau du service mesh pour les communications gRPC.
Définir des DTOs (Data Transfer Objects) distincts par endpoint, exposant uniquement les propriétés strictement nécessaires dans chaque réponse.
Valider le payload entrant contre un schéma JSON strict avec `additionalProperties: false` pour rejeter les propriétés non autorisées lors des requêtes de modification.
Mapper explicitement les champs autorisés vers le modèle métier et désactiver tout binding automatique non maîtrisé du framework (Spring Boot).
Restreindre la modification aux seules propriétés explicitement autorisées pour le rôle de l'appelant via des contrôles d'autorisation granulaires.
Auditer les réponses API avec un outil de diff à chaque déploiement pour détecter les propriétés exposées par inadvertance.
Vérifier systématiquement les autorisations pour chaque requête accédant à une ressource, en s'assurant que l'utilisateur est le propriétaire ou a les droits d'accès requis.
Ne jamais faire confiance aux identifiants d'objets fournis par le client sans validation côté serveur.
Utiliser des identifiants non prévisibles comme des UUID (Universally Unique Identifiers) pour les ressources sensibles plutôt que des IDs séquentiels ou facilement énumérables.
Implémenter une autorisation basée sur les rôles (RBAC) ou les attributs (ABAC) pour contrôler l'accès aux ressources.
Journaliser tous les accès aux ressources sensibles et corréler les événements vers un SIEM pour détecter les comportements anormaux.
Forcer l’utilisation de HTTPS (TLS 1.3 minimum) pour toutes les communications entre le frontend et le backend.
Implémenter une authentification mutuelle TLS (mTLS) pour les communications gRPC entre microservices, assurant l'authentification bidirectionnelle des services.
Valider strictement les certificats côté client et serveur pour toutes les communications chiffrées (y compris vers Azure OpenAI), en utilisant le 'certificate pinning' si applicable.
