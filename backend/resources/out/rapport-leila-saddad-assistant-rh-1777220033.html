L'application **Assistant RH** est une solution web interne de niveau critique moyen, conçue pour automatiser les réponses aux questions RH (congés, politiques, démarches) via un chatbot en lecture seule, sans capacité d'action externe. Architecturée en microservices trois tiers (frontend React, backend Spring Boot, bases de données dédiées par service), elle s'appuie sur une base PostgreSQL on-premise pour les données sensibles (documents internes) et une extension pgvector pour stocker les embeddings locaux utilisés par le RAG. Les microservices communiquent entre eux via gRPC, tandis que le frontend interagit avec le backend via des API REST, sans échange avec des systèmes externes. Le modèle LLM (Azure OpenAI, non fine-tuné) est consommé en mode API externe et enrichi par le RAG via pgvector, sans appel à des outils ou fonctions externes. Aucune persistance centralisée des logs n'est implémentée, et l'application ne gère ni uploads de fichiers ni envoi d'emails, limitant les flux aux interactions utilisateur (navigateur) et aux requêtes internes entre services. Les composants critiques pour le *threat modeling* incluent le chatbot LLM, le moteur RAG avec pgvector, les communications gRPC entre microservices, et l'accès aux données sensibles stockées localement.

Insecure Communication Channels Un attaquant capture une requête HTTPS authentifiée (par exemple, un appel REST ou gRPC) et la rejoue pour déclencher une action non autorisée (par exemple, accéder à un document) via un jeton expiré mais accepté par le système.
Inter-Agent Communication Poisoning L’attaquant compromet le canal ou la passerelle utilisée pour transporter les messages gRPC entre microservices, puis intercepte, modifie, bloque, duplique ou reroute ces messages avant leur réception.
Inter-Agent Communication Poisoning L’attaquant injecte, ajoute ou modifie des champs structurés dans un message gRPC inter-microservices – comme l'intent, la cible, la priorité ou les contraintes – afin de fausser son interprétation ou son exécution par le microservice receveur.
Inter-Agent Communication Poisoning L’attaquant rejoue un message gRPC précédemment valide, ou le réinjecte dans un ordre non prévu, afin de provoquer une exécution redondante, obsolète, hors séquence ou incohérente avec l’état courant du workflow (par exemple, une double validation d'accès à un document).
Jwt manipulation Un attaquant modifie l'en-tête du JWT pour désactiver la vérification de la signature (`alg:none`) ou pour forcer l'utilisation d'une clé publique connue, puis forge un token valide contenant des privilèges accrus.
Jwt manipulation Un attaquant modifie le payload du JWT pour changer son identifiant utilisateur (`sub`) ou ses rôles (`roles`) afin d'obtenir des privilèges administratifs ou d'accéder à des données d'autres utilisateurs.
Jwt manipulation Un attaquant exploite une mauvaise gestion des clés de signature (par exemple, une clé faible, exposée ou non rotative) pour signer ses propres requêtes malveillantes avec un JWT forgé.
Long-term Memory Poisoning L’attaquant dépose un document malveillant (contenant des instructions cachées ou des informations falsifiées) dans une source indexée par le moteur RAG. Lors d’une recherche de contexte, le chatbot récupère ce document et l’intègre à son traitement comme une donnée de référence, altérant ses futures réponses.
Long-term Memory Poisoning L’attaquant injecte dans la conversation un contenu présenté comme une information à retenir. Lors de la génération du résumé de session (s'il y a des mécanismes de 'write back' vers la mémoire long terme), le chatbot reprend ce contenu puis l’enregistre dans sa mémoire long terme pgvector.
Long-term Memory Poisoning L’attaquant exploite un défaut d’authentification ou d’autorisation sur l’API d'accès à la base pgvector, il accède directement aux enregistrements stockés puis modifie les vecteurs, les métadonnées et les entrées de mémoire à la source.
Long-term Memory Poisoning L’attaquant injecte un contenu malveillant qui n’est pas écrit immédiatement en mémoire, mais qui est intégré plus tard lors d’un processus de consolidation, de déduplication, de re-ranking, de re-embedding, de summarization périodique ou de promotion de la mémoire à court terme vers la mémoire à long terme (pgvector).
Man-in-the-Middle Un attaquant intercepte le trafic réseau (par exemple, sur le réseau interne ou entre les microservices via gRPC si non sécurisé) pour lire les informations RH échangées en clair.
Man-in-the-Middle Un attaquant se fait passer pour le serveur backend ou un microservice légitime, ou pour le client (frontend), en présentant un certificat falsifié, pour intercepter les communications.
Man-in-the-Middle Un attaquant intercepte et modifie les données RH échangées entre le client et le serveur ou entre les microservices avant qu'elles n'atteignent leur destination, altérant les requêtes ou les réponses.
ML model evasion Un attaquant fournit des requêtes au chatbot qui, bien que grammaticalement correctes, sont hors du domaine d’entraînement attendu ou contiennent des éléments subtilement modifiés pour provoquer des erreurs de classification ou des réponses inattendues.
ML model evasion Un attaquant manipule une entrée pour qu’elle soit classée (ou comprise par le LLM) comme une catégorie spécifique choisie par l'attaquant, ce qui lui permet d'obtenir un type de réponse qu'il ne devrait pas avoir.
ML model evasion Un attaquant exploite les scores ou les réponses du modèle (par exemple, en observant les variations de comportement) pour affiner ses entrées et améliorer l’évasion, en itérant sur ses requêtes jusqu'à trouver un 'point faible'.
Model denial of service Un attaquant envoie un grand nombre de requêtes simultanées au chatbot, ou soumet des requêtes d'embedding coûteuses au moteur RAG, pour remplir la file d’attente du modèle et empêcher le traitement des requêtes légitimes.
Model denial of service Un attaquant déclenche de multiples appels coûteux au LLM externe (Azure OpenAI) via le chatbot, ou des opérations intensives sur la base pgvector du RAG, afin de saturer les systèmes connectés ou d'épuiser les quotas d'API.
Model denial of service Un attaquant envoie des entrées au chatbot (prompts ou données RAG) dépassant la taille maximale du contexte autorisé afin de provoquer des erreurs, une dégradation du traitement ou une consommation excessive de ressources.
Model denial of service Un attaquant soumet des entrées spécialement conçues (par exemple, des requêtes très complexes, des documents RAG très longs) pour maximiser le temps de calcul du modèle LLM ou du moteur RAG, et ainsi monopoliser les ressources.
Model stealing Un attaquant utilise les sorties du modèle (par exemple, les réponses du chatbot basées sur RAG) pour reconstruire des données sensibles utilisées lors de l’entraînement du RAG.
Model stealing Un attaquant envoie un grand nombre de requêtes au chatbot et au moteur RAG pour apprendre progressivement leur comportement et reconstruire la logique sous-jacente ou la base de connaissances.
Model stealing Un attaquant analyse les réponses du modèle pour déduire ses hyperparamètres internes ou sa structure (applicable au modèle d'embedding local).
Model stealing Un attaquant accède directement au modèle (fichier, stockage, mémoire) de l'embedding local ou à la base pgvector pour les copier.
Model stealing Un attaquant utilise un modèle public similaire pour faciliter l’extraction du modèle d'embedding cible.
Model stealing Un attaquant exploite les scores/probabilités retournés par le modèle pour reconstruire plus précisément sa logique (si le RAG ou l'embedding expose de tels détails).
Model stealing Un attaquant analyse les réponses du modèle pour déduire son architecture (type, couches) du modèle d'embedding local.
Model stealing Un attaquant entraîne un 'modèle shadow' pour imiter le modèle d'embedding cible et analyser son comportement.
Model stealing Un attaquant exploite des informations indirectes (temps de réponse, consommation de ressources) pour extraire des informations sur le modèle d'embedding.
Model Tampering Un attaquant remplace le modèle d'embedding légitime par un modèle modifié dans le Model Registry ou le pipeline de déploiement du RAG.
Model Tampering Un attaquant modifie directement les poids du modèle d'embedding ou les 'checkpoints' stockés, ou altère les données brutes dans la base pgvector.
Model Tampering Un attaquant injecte une modification dans le modèle d'embedding ou le RAG pour qu’il réagisse de manière spécifique à un 'trigger' caché (par exemple, un mot-clé ou une phrase spécifique dans un document) en divulguant des informations ou en adoptant un comportement indésirable.
Model Tampering Un attaquant force le retour vers une ancienne version vulnérable du modèle d'embedding ou de la base pgvector, permettant l'exploitation de failles connues.
Password Reset takeover Un attaquant génère ou devine un token de réinitialisation car celui-ci est court, non aléatoire ou basé sur des valeurs prévisibles (par exemple, ID utilisateur ou timestamp).
Password Reset takeover Un attaquant modifie la requête de reset (par exemple, l'adresse email ou l'en-tête Host) pour que le lien de réinitialisation soit envoyé vers l’attaquant ou un domaine qu’il contrôle, plutôt qu'à l'utilisateur légitime.
Password Reset takeover Un attaquant envoie directement la requête de changement de mot de passe sans que le token de réinitialisation soit correctement vérifié ou validé côté serveur, contournant ainsi le processus de réinitialisation.
Race Condition Un attaquant envoie plusieurs requêtes simultanées pour modifier une donnée RH (par exemple, un statut de congé ou une information utilisateur) avant que le système ne soit mis à jour correctement, créant un état incohérent ou obtenant un avantage indu.
Race Condition Un attaquant exploite un décalage entre la vérification des droits et l'exécution de l'action (vulnérabilité TOCTOU - Time-of-Check to Time-of-Use) pour accéder à une ressource protégée juste après qu'une vérification ait eu lieu mais avant que les accès ne soient révoqués ou mis à jour.
Race Condition Un attaquant déclenche des opérations concurrentes provoquant des états incohérents dans la base de données PostgreSQL ou entre les microservices, menant à des erreurs applicatives ou à des accès non autorisés.
Server-Side Request Forgery ( SSRF ) Un attaquant fournit une URL malveillante (par exemple, dans le prompt du chatbot ou une donnée RAG) qui est ensuite utilisée par le serveur backend pour envoyer des requêtes vers d'autres systèmes internes (par exemple, d'autres microservices, la base PostgreSQL).
Server-Side Request Forgery ( SSRF ) Un attaquant cible les endpoints de métadonnées cloud (par exemple, `169.254.169.254` si l'application est déployée dans un environnement cloud) pour extraire des informations d'identification ou des secrets.
Server-Side Request Forgery ( SSRF ) Un attaquant force le serveur à appeler des services internes (par exemple, `localhost` ou des adresses IP privées) pour interagir avec des services qui ne devraient pas être accessibles depuis l'extérieur.
Server-Side Template Injection Un attaquant injecte une expression malveillante dans une entrée utilisateur (par exemple, un champ de personnalisation d'un rapport) qui est ensuite traitée par le moteur de template du backend pour exécuter du code côté serveur.
Server-Side Template Injection Un attaquant exploite le moteur de template pour accéder à des variables internes du serveur (par exemple, des paramètres de configuration de Spring Boot, des clés API, des variables d'environnement) qui ne devraient pas être exposées.
Server-Side Template Injection Un attaquant injecte du code pour modifier le contenu généré par le template (par exemple, altérer un rapport RH ou une notification) pour afficher des informations trompeuses ou malveillantes.
Session Fixation Un attaquant impose un ID de session prédéfini à la victime (par exemple, via un lien malveillant ou en pré-configurant un cookie), puis attend qu’elle se connecte à l'application 'Assistant RH'. Short-term Memory Poisoning L'attaquant injecte du contenu malveillant dans les segments de la mémoire de travail du chatbot, comme le 'scratchpad' ou les résultats intermédiaires du RAG, influençant directement la logique de la réponse suivante.
Short-term Memory Poisoning L'attaquant insère des instructions persistantes ou des faits falsifiés dans une conversation longue avec le chatbot pour forcer leur inclusion lors de la phase de résumé ou de compression du contexte, altérant ainsi la compréhension du LLM pour les requêtes ultérieures de la session.
SQL injection Un attaquant injecte du code SQL dans un formulaire de connexion ou un paramètre d'API pour se connecter sans identifiants valides à l'application 'Assistant RH'.
SQL injection Un attaquant exploite une vulnérabilité SQL Injection pour extraire des informations RH sensibles (par exemple, des données personnelles des employés, des politiques internes, des documents confidentiels) de la base de données PostgreSQL.
SQL injection Un attaquant utilise une SQL Injection pour exécuter des commandes système sur le serveur de base de données PostgreSQL (si les privilèges le permettent), obtenant un accès shell ou un contrôle du serveur.
SQL injection Un attaquant injecte des commandes SQL pour modifier ou supprimer des données RH cruciales dans la base de données PostgreSQL, compromettant l'intégrité et la disponibilité des informations.
User Impersonation L’attaquant obtient les identifiants ou facteurs d’authentification d’un utilisateur légitime (par exemple, par phishing ou vol de 'credentials'), puis ouvre une session auprès de l’application ou du service d’orchestration sous cette identité afin de soumettre des demandes comme s’il était cet utilisateur.
User Impersonation L’attaquant récupère un cookie de session, un jeton de session ou un snapshot de contexte associé à une session déjà authentifiée, puis le réutilise pour reprendre l’interaction avec le chatbot sous l’identité de l’utilisateur légitime.
User Impersonation L’attaquant injecte dans le contexte de l’application ou du chatbot une assertion d’identité, de rôle ou d’autorité utilisateur (par exemple, dans un prompt, une mémoire ou un message inter-service) puis amène le chatbot à traiter cette déclaration comme une preuve valide sans vérification indépendante de l'authentification.
User Impersonation L’attaquant vole ou forge un token de délégation (par exemple, un JWT si utilisé) utilisé par l’application pour agir au nom d’un utilisateur auprès de services tiers (bien que le LLM n'appelle pas d'outils externes, d'autres microservices pourraient le faire), puis l’utilise pour déclencher des actions comme si elles provenaient de cet utilisateur.
WebSocket Exploitation Un attaquant force le navigateur de la victime à établir une connexion WebSocket vers l'application 'Assistant RH' en utilisant ses cookies de session (Cross-Site WebSocket Hijacking - CSWSH).
WebSocket Exploitation Un attaquant intercepte et modifie les messages WebSocket échangés entre le client (frontend) et le serveur (backend) du chatbot pour altérer les questions ou les réponses.
WebSocket Exploitation Un attaquant ouvre un grand nombre de connexions WebSocket persistantes pour épuiser les ressources du serveur backend, provoquant un déni de service pour le chatbot.
