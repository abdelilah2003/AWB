Description de l'Application

Nom
assistant RH
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'application "Assistant RH" est une solution web interne de niveau critique medium, accessible via navigateur, conçue pour automatiser les réponses aux questions RH (congés, politiques internes, démarches administratives) en exploitant une architecture trois tiers en microservices. Le frontend React interagit avec un backend Spring Boot structuré en services autonomes, chacun disposant de sa propre base de données relationnelle AWS RDS MySQL hébergée dans le cloud AWS, avec des échanges inter-services sécurisés via gRPC. Pour les fonctionnalités d'IA, l'application intègre un chatbot en lecture seule basé sur Azure OpenAI (non fine-tuné), exploitant une architecture RAG avec des embeddings locaux stockés dans une base vectorielle pgvector sur PostgreSQL, sans capacité à déclencher des actions externes. Les données traitées, de sensibilité élevée (données personnelles), transitent uniquement en interne sans upload de fichiers ni envoi d'emails, tandis que les logs restent décentralisés. Les composants critiques pour le threat modeling incluent le flux utilisateur-frontend, les communications gRPC entre microservices, l'intégration avec l'API externe Azure OpenAI, et les accès concurrentiels aux bases MySQL et pgvector, sans mécanisme de broker ou de tâches asynchrones. L'absence de centralisation des logs et de chiffrement explicite des données au repos ou en transit constitue des points d'attention majeurs pour la sécurité.

Menace Description
AI supply chain tampering Consiste à compromettre un composant de la chaîne d’approvisionnement de l'application Assistant RH, tels que les bibliothèques d'embeddings, les dépendances du RAG ou les sources de données utilisées pour alimenter la base vectorielle pgvector, afin d’introduire un comportement malveillant ou d'altérer la qualité des réponses du chatbot.
Broken Authentication Les mécanismes d'authentification de l'application Assistant RH, accessible via navigateur par des utilisateurs internes, présentent des failles (brute-force, jeton faible, absence de ré-authentification) permettant la prise de contrôle de comptes et l'accès non autorisé à des données RH sensibles.
Broken Object Level Authorization Les microservices du backend de l'Assistant RH n'appliquent pas de contrôle de propriété granulaire sur chaque ressource accédée. Tout utilisateur interne authentifié pourrait potentiellement lire ou modifier les données personnelles d'autrui (e.g., informations de congé d'un autre employé) en substituant l'identifiant de l'objet dans la requête.
Broken Object Property Level Authorization L'API gRPC des microservices de l'Assistant RH expose des propriétés d'objets sensibles dans ses réponses ou accepte leur modification sans contrôle granulaire au niveau du champ, permettant à un attaquant d'accéder ou de modifier des attributs de données personnelles qui ne devraient pas être exposés ou manipulables.
Bruteforce Consiste à tenter d’accéder à des comptes d'utilisateurs internes de l'Assistant RH ou des secrets d'API des microservices en testant systématiquement toutes les combinaisons possibles jusqu’à trouver la bonne, ce qui est facilité par l'accès web interne et la nature critique des données RH.
Buffer Manipulation Consiste à manipuler un buffer mémoire en fournissant plus de données que sa capacité aux microservices du backend Spring Boot ou aux composants gRPC de l'Assistant RH, entraînant l’écrasement de la mémoire adjacente et un comportement imprévu du programme, potentiellement avec élévation de privilèges ou fuite de données.

Cross-Site Scripting ( XSS ) Consiste à injecter du code JavaScript malveillant dans le frontend React de l'application Assistant RH à travers des entrées utilisateur non filtrées, telles que les noms d'utilisateurs ou les contenus de chat, permettant d'exécuter du code dans le navigateur des utilisateurs internes.
Data Poisoning Bien que le LLM Azure OpenAI ne soit pas fine-tuné par l'application Assistant RH, la menace de Data Poisoning est pertinente pour les données utilisées dans l'architecture RAG. Cela consiste à manipuler intentionnellement les données stockées dans la base vectorielle pgvector sur PostgreSQL (embeddings locaux) afin d’altérer le comportement, les performances ou les décisions du chatbot.
deserialization injection Consiste à exploiter la désérialisation de données non fiables dans le backend Spring Boot de l'Assistant RH (e.g., via des requêtes gRPC malveillantes ou des données stockées) pour manipuler des objets ou exécuter du code malveillant sur les microservices.
Direct prompt injection Consiste à injecter des instructions malveillantes directement dans un prompt soumis par un utilisateur au chatbot de l'Assistant RH afin de manipuler le comportement du modèle Azure OpenAI et de contourner ses règles de sécurité ou ses objectifs initiaux.

HTTP Request Smuggling Consiste à exploiter une désynchronisation entre un proxy frontal (e.g., load balancer, API Gateway) et le backend Spring Boot des microservices de l'Assistant RH pour injecter une requête HTTP cachée, permettant de contourner les contrôles de sécurité et d'accéder à des ressources internes.
IDOR exploitation Consiste à exploiter une vulnérabilité d'IDOR (Insecure Direct Object Reference) dans l'application Assistant RH en manipulant des identifiants (ID de congés, d'utilisateurs, de documents RH) pour accéder à des ressources ou des actions sans autorisation, particulièrement critique avec des données personnelles sensibles.
Indirect prompt injection Consiste à injecter des instructions malveillantes dans une source externe (un document ou une donnée stockée) que le RAG de l'Assistant RH va ensuite lire, vectoriser, et utiliser indirectement pour alimenter le chatbot basé sur Azure OpenAI, manipulant ainsi les réponses du LLM à l'insu de l'utilisateur.

Model stealing Un attaquant envoie un grand nombre de requêtes au chatbot de l'Assistant RH pour apprendre progressivement le comportement de la logique RAG et des embeddings locaux, dans le but de reconstruire un modèle similaire.
Model stealing Un attaquant analyse les réponses du chatbot pour déduire les hyperparamètres internes ou la structure de la base d'embeddings locaux.
Model stealing Un attaquant accède directement à la base vectorielle pgvector (fichier, stockage, mémoire) pour copier les embeddings locaux ou des informations sur leur génération.
Model stealing Un attaquant exploite les informations indirectes (temps de réponse, consommation de tokens) de l'API Azure OpenAI pour extraire des informations sur le modèle distant.
Oauth Abuse Un attaquant modifie le `redirect_uri` dans la requête OAuth pour rediriger le code d'autorisation ou le token vers son propre serveur, prenant ainsi le contrôle de la session de la victime sur l'Assistant RH.
Oauth Abuse Un attaquant envoie un lien OAuth piégé à la victime pour que l'application Assistant RH associe la session de la victime au compte de l'attaquant, permettant une prise de contrôle.
Oauth Abuse Un attaquant récupère un token OAuth exposé (e.g., dans une URL, un log mal protégé ou le navigateur) et l'utilise pour accéder à l'application au nom de la victime.
Password Reset takeover Un attaquant génère ou devine un token de réinitialisation car celui-ci est court, non aléatoire ou basé sur des valeurs prévisibles, et l'utilise pour réinitialiser le mot de passe d'un utilisateur de l'Assistant RH.
Password Reset takeover Un attaquant modifie la requête de reset (ex: en manipulant le champ email ou l'en-tête Host) pour que le lien de réinitialisation soit envoyé vers une adresse contrôlée par l’attaquant.
Password Reset takeover Un attaquant envoie directement la requête de changement de mot de passe sans que le token ne soit correctement vérifié côté serveur, permettant de contourner le processus de réinitialisation.
Race Condition Un attaquant envoie plusieurs requêtes simultanées à un microservice pour modifier une donnée RH (ex: statut de congé) avant que le système ne soit mis à jour correctement, créant un état incohérent.
Race Condition Un attaquant exploite un décalage entre une vérification d'autorisation et l'exécution de l'action (Time-of-Check to Time-of-Use - TOCTOU) pour accéder à une ressource protégée avant que les permissions ne soient complètement appliquées.
Race Condition Un attaquant déclenche des opérations concurrentes provoquant des états incohérents ou des erreurs dans les bases de données MySQL ou pgvector, par exemple en modifiant simultanément un solde de congés.
Server-Side Request Forgery ( SSRF ) Un attaquant manipule une URL ou un paramètre d'entrée pour forcer le backend de l'Assistant RH à envoyer des requêtes vers d’autres microservices internes ou des bases de données MySQL/pgvector (e.g., via des appels gRPC internes) qui ne devraient pas être directement accessibles.
Server-Side Request Forgery ( SSRF ) Un attaquant cible les endpoints de métadonnées cloud (AWS EC2 metadata service) pour récupérer des identifiants temporaires AWS ou des informations de configuration sensibles du serveur hôte.
Server-Side Request Forgery ( SSRF ) Un attaquant force le serveur à appeler des services internes non exposés (localhost, réseau privé) pour collecter des informations ou déclencher des actions.
Server-Side Template Injection Un attaquant injecte une expression malveillante dans un champ d'entrée qui est ensuite traité par un moteur de template côté serveur (ex: ' ${7*7} ' pour une exécution de code) de l'Assistant RH.
Server-Side Template Injection Un attaquant exploite le moteur de template pour accéder à des variables internes (configurations, clés API pour Azure OpenAI, identifiants BDD) ou aux objets Java du serveur.
Server-Side Template Injection Un attaquant injecte du code pour modifier le contenu généré par le template, affichant des informations RH falsifiées ou des messages trompeurs sur l'interface.
Session Fixation Un attaquant impose un ID de session prédéfini à la victime (via URL, cookie non sécurisé) avant qu'elle ne se connecte à l'Assistant RH, puis attend qu’elle s’authentifie.
Session Fixation Un attaquant envoie un lien contenant un ID de session prédéfini à la victime de l'Assistant RH, qui est ensuite utilisé lors de sa connexion.
Short-term Memory Poisoning L'attaquant injecte du contenu malveillant dans les segments de la mémoire de travail de l'agent (chatbot), comme le scratchpad ou les résultats d'API intermédiaires, altérant le raisonnement du LLM pour la réponse immédiate.
Short-term Memory Poisoning L'attaquant insère des instructions persistantes ou des faits falsifiés dans une conversation longue avec le chatbot pour forcer leur inclusion lors de la phase de résumé (compression) du contexte, influençant les réponses futures de la session.
SQL injection Un attaquant injecte du code SQL dans un formulaire de connexion ou un paramètre API de l'Assistant RH pour se connecter à la base de données MySQL ou pgvector sans identifiants valides.
SQL injection Un attaquant exploite une vulnérabilité SQL Injection pour extraire des informations sensibles (ex: listes d'employés, salaires, politiques internes) de la base de données.
SQL injection Un attaquant utilise une SQL Injection pour modifier ou supprimer des données dans les bases MySQL ou pgvector (ex: modifier des dates de congés, des informations personnelles des employés).
SQL injection Un attaquant injecte des commandes SQL pour tenter d'exécuter des commandes système sur le serveur de base de données (si les privilèges le permettent).
User Impersonation L’attaquant obtient les identifiants ou facteurs d’authentification d’un utilisateur légitime, puis ouvre une session auprès de l’Assistant RH sous cette identité pour soumettre des demandes comme s’il était cet utilisateur.
User Impersonation L’attaquant récupère un cookie de session, un jeton de session ou un snapshot de contexte associé à une session déjà authentifiée, puis le réutilise pour reprendre l’interaction avec l’Assistant RH sous l’identité de l’utilisateur légitime.
User Impersonation L’attaquant injecte dans le contexte de l’application ou du chatbot une assertion d’identité, de rôle ou d’autorité utilisateur (dans un prompt, une mémoire, un message inter-service ou un output d'API) puis amène l’application à traiter cette déclaration comme une preuve valide sans vérification indépendante.
User Impersonation L’attaquant vole ou forge un token (ex: JWT, OAuth) utilisé par l’application pour agir au nom d’un utilisateur auprès de services tiers (ex: une API gRPC interne), puis l’utilise pour déclencher des actions comme si elles provenaient de cet utilisateur.
