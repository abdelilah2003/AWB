Description de l'Application

Nom
assistant RH
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'application "Assistant RH" est une solution web interne de niveau critique medium, accessible via navigateur, conçue pour répondre aux questions des employés sur les congés, politiques internes et procédures administratives en automatisant les réponses RH courantes. Architecturée en microservices selon un modèle trois tiers (présentation, logique applicative, données), elle repose sur un frontend React communiquant avec un backend Spring Boot via des appels gRPC, chaque microservice disposant de sa propre base de données relationnelle AWS RDS MySQL hébergée dans le cloud AWS. Les données manipulées, de sensibilité élevée (données personnelles), sont isolées par service, tandis que les microservices intercommuniquent exclusivement via gRPC sans broker ni centralisation des logs. Le système intègre un module d'IA générative en lecture seule, basé sur Azure OpenAI (non fine-tuné) consommé via API externe, couplé à une architecture RAG utilisant des embeddings locaux et une base vectorielle pgvector sur PostgreSQL pour enrichir les réponses du chatbot sans accès à des outils externes. Les flux principaux incluent les interactions utilisateur-frontend, les échanges gRPC entre microservices, et les requêtes du moteur RAG vers la base vectorielle et le LLM externe, sans persistance des échanges ni upload de fichiers. L'absence de tâches asynchrones, de notifications email ou de composants tiers (hors LLM) simplifie le périmètre fonctionnel, mais impose une gestion stricte des accès aux données sensibles et des communications inter-services pour le threat modeling.

Menaces Retenues du Catalogue

Menaces issues du catalogue et retenues selon le contexte technique et fonctionnel de l application.
Menace Description
SQL injection L'attaquant injecte des requêtes SQL malveillantes via les entrées utilisateur (frontend ou gRPC) pour manipuler les bases de données AWS RDS MySQL ou la base vectorielle pgvector de l'application "Assistant RH". Cela peut cibler les questions des employés, les recherches dans les politiques internes ou les procédures administratives.
NoSQL injection L'attaquant injecte des requêtes ou des objets malveillants via le frontend ou les appels gRPC, ciblant la base vectorielle pgvector (qui est une extension PostgreSQL mais peut être vulnérable à des logiques d'injection similaires) de l'application "Assistant RH" pour manipuler les requêtes de recherche d'embeddings ou les conditions d'accès.
Command injection Un attaquant injecte des commandes système malveillantes via les entrées utilisateur ou les communications gRPC, que les microservices Spring Boot de l'application "Assistant RH" pourraient exécuter à leur insu.
Cross-Site Scripting ( XSS ) L'attaquant injecte du code JavaScript malveillant dans l'interface utilisateur React de l'application "Assistant RH" via des entrées utilisateur non filtrées (par exemple, dans les questions du chatbot ou les données affichées), qui sera exécuté dans le navigateur des autres employés.
Cross-Site Request Forgery L'attaquant force un employé authentifié de l'application "Assistant RH" à exécuter une action non désirée (par exemple, changer ses préférences, consulter une information spécifique) à son insu, en exploitant la confiance du navigateur envers l'application.
Bruteforce L'attaquant tente d'accéder à un compte utilisateur légitime de l'application "Assistant RH" en testant systématiquement un grand nombre de combinaisons d'identifiants et de mots de passe sur l'interface de connexion.
Directory Traversal L'attaquant manipule des chemins de fichiers via des entrées utilisateur ou des requêtes gRPC pour accéder à des fichiers ou répertoires en dehors du répertoire autorisé sur les serveurs des microservices Spring Boot de l'application "Assistant RH".
Distributed Denial of Service L'attaquant submerge l'application "Assistant RH" (frontend, backend, LLM externe) avec un grand volume de trafic ou de requêtes complexes, provenant de multiples sources, afin de la rendre indisponible pour les employés internes.
Server-Side Request Forgery ( SSRF ) L'attaquant exploite une mauvaise validation des entrées pour forcer un microservice Spring Boot de l'application "Assistant RH" à envoyer des requêtes HTTP vers des ressources internes (autres microservices, bases de données) ou externes (métadonnées AWS, API Azure OpenAI) avec les privilèges du microservice compromis.
Fuzzing L'attaquant envoie des entrées aléatoires ou malformées aux microservices ou à l'interface du chatbot de l'application "Assistant RH" afin de provoquer des comportements anormaux, des plantages, ou d'identifier des vulnérabilités non découvertes.
deserialization injection L'attaquant exploite la désérialisation de données non fiables (par exemple, des payloads gRPC, des données de session) au sein des microservices Spring Boot de l'application "Assistant RH" pour manipuler des objets ou exécuter du code malveillant.
Race Condition L'attaquant exploite un problème de concurrence dans l'application "Assistant RH", où plusieurs requêtes ou actions critiques sont traitées simultanément par les microservices, pouvant conduire à des états incohérents ou à des contournements de sécurité.
Session Fixation L'attaquant force un employé de l'application "Assistant RH" à utiliser un identifiant de session connu par l'attaquant, puis prend le contrôle de la session de l'employé après son authentification réussie.
Man-in-the-Middle L'attaquant intercepte et éventuellement modifie les communications (utilisateur-frontend, inter-microservices gRPC, backend-Azure OpenAI, backend-DB) de l'application "Assistant RH" sans que les parties s'en rendent compte.
Jwt manipulation Si l'application "Assistant RH" utilise des JSON Web Tokens (JWT) pour la gestion de session ou l'autorisation inter-microservices, un attaquant pourrait exploiter des failles dans leur gestion (signatures faibles, algorithmes non sécurisés, claims non validés) pour usurper des identités ou élever des privilèges.
IDOR exploitation L'attaquant manipule les identifiants d'objet (ID) dans les requêtes de l'application "Assistant RH" (frontend ou gRPC) pour accéder à des ressources (informations RH, conversations, etc.) sans autorisation appropriée, en exploitant l'absence de contrôles de propriété côté serveur.
Client side template injection L'attaquant injecte du code malveillant dans le moteur de template côté client (React) de l'application "Assistant RH" via des entrées utilisateur (par exemple, des messages du chatbot), afin d'exécuter du JavaScript ou de manipuler l'interface utilisateur dans le navigateur des victimes.
Information Disclosure L'application "Assistant RH" expose des informations sensibles (données personnelles, secrets, configuration interne) à des employés non autorisés, en raison d'une mauvaise configuration, de messages d'erreur détaillés, ou de logs non centralisés.
Password Reset takeover L'attaquant exploite des failles dans le mécanisme de réinitialisation de mot de passe de l'application "Assistant RH" pour prendre le contrôle du compte d'un employé sans disposer de ses identifiants.
Data Poisoning L'attaquant manipule intentionnellement les données utilisées par l'architecture RAG de l'application "Assistant RH" (embeddings locaux, base vectorielle pgvector) afin d'altérer le comportement du chatbot, ses performances ou ses décisions lors de la génération de réponses RH.
Model stealing L'attaquant tente d'accéder, copier, extraire ou reconstruire le modèle LLM d'Azure OpenAI (via l'API) ou le modèle RAG interne (corpus, embeddings) de l'application "Assistant RH" sans autorisation, pour exploiter sa propriété intellectuelle ou créer un modèle équivalent.
AI supply chain tampering L'attaquant compromet un composant de la chaîne d'approvisionnement de l'IA de l'application "Assistant RH" (par exemple, les bibliothèques d'embeddings locales, le pipeline de mise à jour du corpus RAG, les dépendances des microservices ou les points d'accès à Azure OpenAI) afin d'introduire un comportement malveillant.
Direct prompt injection L'attaquant injecte des instructions malveillantes directement dans le prompt du chatbot de l'application "Assistant RH" afin de manipuler le comportement du LLM (Azure OpenAI) et de contourner ses garde-fous de sécurité.
Indirect prompt injection L'attaquant injecte des instructions malveillantes dans une source externe (le corpus de documents RH utilisé pour le RAG) que le chatbot "Assistant RH" va ensuite lire et exécuter indirectement lors de la génération d'une réponse.
Model denial of service L'attaquant exploite les limites computationnelles ou économiques du chatbot "Assistant RH" et de son LLM externe (Azure OpenAI) en générant une charge excessive ou anormale (requêtes complexes, grands prompts) afin de dégrader ses performances, épuiser ses ressources ou provoquer son indisponibilité.
ML model evasion L'attaquant manipule les entrées du chatbot "Assistant RH" afin de contourner ses prédictions ou décisions lors de l'inférence avec Azure OpenAI, dans le but d'obtenir des réponses indésirables ou de contourner les garde-fous.
Model Tampering L'attaquant compromet l'intégrité du système RAG de l'application "Assistant RH" en modifiant la base vectorielle pgvector, les embeddings locaux, ou le processus de mise à jour du corpus RH, afin d'introduire un biais ou un comportement malveillant persistant dans les réponses du chatbot.
Long-term Memory Poisoning L'attaquant injecte ou altère des informations malveillantes dans la mémoire persistante de l'Assistant RH (la base vectorielle pgvector du RAG, qui contient le corpus RH) afin d'influencer durablement le comportement du chatbot.
Short-term Memory Poisoning L'attaquant injecte ou altère des informations malveillantes dans la mémoire temporaire (contexte de conversation) du chatbot "Assistant RH" afin d'influencer son état de travail, son raisonnement ou ses décisions pendant la session en cours.
User Impersonation L'attaquant usurpe l'identité d'un employé légitime de l'application "Assistant RH" pour interagir avec le chatbot ou les microservices, en exploitant des failles d'authentification ou de session.

Mitigations

Mitigation à appliquer
Utiliser exclusivement des requêtes préparées ou des ORM sécurisés (comme ceux de Spring Boot) pour interagir avec AWS RDS MySQL et pgvector.
Valider strictement toutes les entrées utilisateur côté backend (type, format, taille, contenu) avant de les inclure dans des requêtes aux bases de données.
Appliquer le principe du moindre privilège aux comptes d'accès aux bases de données, en limitant les droits aux opérations strictement nécessaires pour chaque microservice.
Chiffrer les données sensibles au repos dans AWS RDS MySQL et pgvector.
Ne jamais exposer la base de données sur Internet ; la placer dans un sous-réseau privé (VPC AWS) et restreindre l'accès via des Security Groups AWS aux seuls microservices autorisés.
Activer les logs natifs du fournisseur cloud (AWS) pour la base de données afin de surveiller les requêtes suspectes.
Utiliser des requêtes paramétrées pour toutes les interactions avec la base vectorielle pgvector.
Valider et nettoyer strictement toutes les entrées utilisateur avant de les passer au moteur de recherche RAG ou à la base vectorielle.
Implémenter des contrôles d'accès granulaires au niveau de l'application sur les données récupérées par le RAG pour s'assurer que l'utilisateur est autorisé à consulter l'information.
Restreindre les privilèges du microservice RAG sur la base pgvector au strict nécessaire (lecture uniquement, sur des colonnes spécifiques).
Ne jamais construire des commandes système via concaténation de chaînes avec des entrées utilisateur. Utiliser des API natives ou des bibliothèques sécurisées qui évitent l'interprétation shell.
Valider strictement toutes les entrées utilisateur (via une liste blanche) avant de les utiliser dans des opérations potentiellement risquées.
Appliquer le principe du moindre privilège pour les processus des microservices (exécuter avec des comptes non privilégiés, restreindre les droits d'accès aux binaires système).
Isoler les environnements d’exécution des microservices (conteneurs) et limiter leurs capacités.
Implémenter un mécanisme d'échappement systématique des données en sortie (HTML, JavaScript, URL) avant de les afficher dans le frontend React.
Valider et filtrer strictement toutes les entrées utilisateur côté serveur (backend Spring Boot).
Implémenter une politique de sécurité de contenu (CSP) stricte pour le frontend afin de restreindre l’exécution de scripts non autorisés.
Activer les attributs de sécurité des cookies (HttpOnly, Secure, SameSite=Strict).
Implémenter des tokens anti-CSRF synchronizer (CSRF tokens) pour toutes les requêtes qui modifient l'état de l'application.
Implémenter des cookies de session avec l'attribut SameSite=Strict par défaut.
Vérifier l'en-tête Origin et Referer côté serveur pour bloquer les requêtes cross-site non autorisées.
Exiger une re-authentification forte (step-up authentication) pour toute action critique.
Implémenter un rate limiting strict sur les endpoints d'authentification par IP, par compte et par session.
Bloquer temporairement le compte après un nombre défini d'échecs de connexion (account lockout).
Imposer une authentification multifacteur (MFA) robuste pour tous les comptes.
Implémenter un CAPTCHA adaptatif après détection de comportement suspect.
Journaliser toutes les tentatives d’authentification (succès/échec) et surveiller les comportements anormaux.
Implémenter une validation stricte des chemins (via une liste blanche) et ne jamais faire confiance aux entrées utilisateur.
Utiliser des identifiants indirects au lieu de chemins de fichiers directs lorsque cela est possible.
Stocker les fichiers sensibles (configurations, logs, secrets) en dehors des répertoires accessibles par l'application web.
Appliquer le principe du moindre privilège sur les accès aux fichiers pour les microservices.
