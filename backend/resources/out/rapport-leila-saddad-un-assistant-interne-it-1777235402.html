Description de l'Application

Nom
un assistant interne IT
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'application est un assistant IT interne de type web (React) accessible uniquement aux employés via navigateur, conçue en architecture N-Tier avec microservices (Spring Boot) communiquant en REST interne. Elle s'authentifie via LDAP directement auprès d'Active Directory pour valider les utilisateurs internes et traite des données sensibles (documents internes) sans upload ni échange d'emails. Chaque microservice possède sa propre base de données relationnelle PostgreSQL hébergée sur AWS RDS, avec une instance dédiée à pgvector pour stocker les embeddings locaux (modèle RAG) et indexer la base de connaissances interne. Le chatbot intègre Azure OpenAI via API externe, interrogeant dynamiquement la base vectorielle pour générer des réponses contextuelles sans appeler d'outils externes ni exposer directement le LLM. Les flux critiques incluent l'authentification LDAP vers AD, les requêtes REST entre microservices, l'accès aux données sensibles dans RDS, et les appels API vers Azure OpenAI, le tout dans un environnement cloud AWS sans broker ni tâches asynchrones. La criticité fonctionnelle moyenne et l'absence d'agents ou de ML hors LLM/RAG simplifient le périmètre, mais les interactions avec AD, le LLM externe et les données sensibles nécessitent une attention particulière pour le threat modeling.

Menace Description
SQL injection Dans l'application d'assistant IT interne, un attaquant injecte des requêtes SQL malveillantes dans une base de données PostgreSQL (y compris pgvector pour les embeddings RAG) via des entrées utilisateur non sécurisées, comme les champs de recherche ou les paramètres de requêtes API.
AI supply chain tampering Dans l'assistant IT interne, un attaquant compromet un composant de la chaîne d’approvisionnement de l'IA, comme les documents internes bruts, les pipelines de génération d'embeddings, ou même la base vectorielle pgvector elle-même, afin d’introduire un comportement malveillant dans la logique RAG.
Broken Authentication Les mécanismes d'authentification de l'assistant IT interne, notamment l'authentification LDAP directe auprès d'Active Directory et la gestion des sessions pour l'accès aux microservices via l'interface web (React), présentent des failles (brute-force, jeton faible, absence de ré-authentification) permettant la prise de contrôle de comptes d'employés.
Broken Object Level Authorization L'API REST des microservices de l'assistant IT interne n'applique pas un contrôle de propriété adéquat sur chaque ressource (documents internes, données utilisateurs) accédée. Ainsi, un employé malveillant ou un attaquant ayant compromis un compte pourrait accéder aux objets (documents, informations) d'autrui en substituant l'identifiant dans la requête API.

Bruteforce Un attaquant tente d’accéder aux comptes des employés de l'assistant IT interne en testant systématiquement toutes les combinaisons possibles d'identifiants et de mots de passe sur l'interface de connexion web, ou en ciblant les APIs d'authentification LDAP vers Active Directory.
Cross-Site Request Forgery Un attaquant tente de forcer un utilisateur (employé) authentifié sur l'assistant IT interne à exécuter une action non désirée, à son insu, en exploitant les vulnérabilités du frontend React ou des APIs REST des microservices.
Cross-Site Scripting ( XSS ) Un attaquant injecte du code JavaScript malveillant dans l'application web React de l'assistant IT via des entrées utilisateur non filtrées (par exemple, champs de recherche, noms de documents, ou même les réponses du LLM si elles sont mal gérées) affichées à d'autres employés.

deserialization injection Un attaquant exploite la désérialisation de données non fiables (par exemple, dans les communications REST entre microservices Spring Boot) pour manipuler des objets applicatifs ou exécuter du code malveillant sur les serveurs des microservices de l'assistant IT interne.
Directory Traversal Dans l'application d'assistant IT interne, un attaquant manipule les chemins de fichiers (par exemple, dans les requêtes d'accès aux 'documents internes' pour le RAG) pour accéder à des fichiers ou répertoires en dehors du répertoire autorisé sur les serveurs des microservices ou les systèmes de stockage d'AWS.
Direct prompt injection Un attaquant injecte des instructions malveillantes directement dans les prompts du chatbot de l'assistant IT interne pour manipuler le comportement du modèle Azure OpenAI, le pousser à ignorer ses directives ou à divulguer des informations sensibles.
Distributed Denial of Service Un attaquant submerge les serveurs des microservices (Spring Boot) et le frontend (React) de l'assistant IT interne avec un grand volume de trafic provenant de multiples sources, rendant l'application indisponible pour les employés. Le backend des microservices peut également être ciblé.
File Inclusion Un attaquant exploite les microservices de l'assistant IT interne pour inclure et exécuter des fichiers locaux ou distants via des entrées utilisateur non sécurisées, potentiellement dans le contexte de la gestion des 'documents internes' pour le RAG.

HTTP Request Smuggling Dans l'architecture N-Tier de l'assistant IT interne avec microservices (Spring Boot) communiquant en REST, un attaquant exploite une désynchronisation entre un proxy frontal (load balancer, WAF) et les microservices backend pour injecter une requête HTTP cachée.
Indirect prompt injection Un attaquant injecte des instructions malveillantes dans les 'documents internes' qui constituent la base de connaissances du système RAG. Le modèle Azure OpenAI de l'assistant IT interne, en interrogeant cette base vectorielle, lit et exécute indirectement ces instructions sans que l'utilisateur n'ait à les saisir directement.
Information Disclosure L'application d'assistant IT interne traite des données sensibles (documents internes, informations d'employés) et est hébergée sur AWS. Une mauvaise configuration du système, des messages d'erreur détaillés, ou des logs non sécurisés pourraient exposer ces informations à des employés non autorisés ou même à des attaquants externes.
Jwt manipulation Les microservices (Spring Boot) de l'assistant IT interne communiquent en REST. Si des JSON Web Tokens (JWT) sont utilisés pour l'authentification et l'autorisation entre ces services, un attaquant pourrait manipuler ces tokens pour usurper l'identité de services légitimes, obtenir des privilèges non autorisés ou accéder à des données sensibles.
LDAP injection L'application d'assistant IT interne s'authentifie directement via LDAP auprès d'Active Directory. Un attaquant injecte des requêtes malveillantes dans le filtre LDAP via des entrées utilisateur non sécurisées (par exemple, dans le champ d'identifiant ou de mot de passe) afin de manipuler les requêtes d’authentification ou de recherche.
Long-term Memory Poisoning L'assistant IT interne utilise une base vectorielle pgvector pour stocker les embeddings locaux de sa base de connaissances interne (documents internes) comme mémoire persistante pour le modèle RAG. Un attaquant injecte ou altère des informations malveillantes dans cette mémoire persistante.
Man-in-the-Middle Un attaquant intercepte et potentiellement modifie les communications entre les composants de l'application d'assistant IT interne : Frontend React et Microservices, Microservices entre eux (REST interne), ou Microservices et Active Directory (LDAP), et Microservices et Azure OpenAI (API externe).
ML model evasion Un attaquant manipule les entrées (prompts) du modèle Azure OpenAI du chatbot de l'assistant IT interne afin de contourner ses prédictions ou décisions lors de l’inférence, le poussant à générer des réponses indésirables ou à ignorer des instructions.
Model denial of service Un attaquant exploite les limites computationnelles, opérationnelles ou économiques du modèle Azure OpenAI (via son API) ou de la couche RAG (base pgvector, microservices) de l'assistant IT interne en générant une charge excessive ou anormale.
Model stealing L'application utilise une logique RAG avec des embeddings locaux stockés dans pgvector pour sa base de connaissances interne. Un attaquant tente d'accéder, copier, extraire ou reconstruire ces embeddings ou la logique RAG sous-jacente sans autorisation, via des failles d’infrastructure, des fuites internes ou des requêtes API, afin de recréer la base de connaissances ou exploiter cette propriété intellectuelle.
Model Tampering L'application d'assistant IT interne utilise une base vectorielle pgvector pour les embeddings de sa base de connaissances. Un attaquant tente de compromettre l’intégrité de cette base de connaissances en modifiant ses embeddings, ses artefacts (documents sources) ou son processus de mise à jour.

Race Condition Dans l'architecture microservices (Spring Boot) de l'assistant IT interne, avec des accès concurrents aux bases de données PostgreSQL et à d'autres ressources partagées, un attaquant exploite un problème de concurrence où plusieurs requêtes modifient une ressource en même temps.
Server-Side Request Forgery ( SSRF ) Dans l'assistant IT interne, un attaquant exploite une mauvaise validation des entrées (par exemple, dans les requêtes de recherche RAG ou les appels à Azure OpenAI) pour forcer un microservice Spring Boot à envoyer des requêtes vers des ressources internes (AWS metadata, autres services internes) ou externes non autorisées avec ses propres privilèges.
Session Fixation Un attaquant tente de forcer un employé de l'assistant IT interne à utiliser un identifiant de session connu par l’attaquant afin de prendre le contrôle de sa session après que l'employé se soit authentifié via LDAP/Active Directory sur l'application web React.
Short-term Memory Poisoning Dans le contexte du chatbot de l'assistant IT interne, un attaquant injecte ou altère des informations malveillantes dans la mémoire temporaire de l'agent (le contexte de la conversation envoyé au LLM Azure OpenAI) afin d’influencer son raisonnement ou ses décisions pendant la session en cours.
User Impersonation Dans le contexte du chatbot de l'assistant IT interne, un attaquant vise à usurper l’identité d’un employé légitime pour faire exécuter des actions par le chatbot (via Azure OpenAI/RAG) en son nom, potentiellement pour accéder à des données sensibles ou manipuler les réponses.

Utiliser exclusivement des requêtes préparées (prepared statements) pour toutes les interactions avec la base de données PostgreSQL.
Interdire la concaténation dynamique de chaînes SQL dans le code des microservices Spring Boot.
Valider strictement toutes les entrées utilisateur côté serveur (type, format, taille et contenu).
Chiffrer les données sensibles au repos dans PostgreSQL RDS et en transit via TLS 1.3.
Appliquer le principe du moindre privilège aux comptes d'accès à la base de données et aux rôles IAM sur AWS RDS.
Implémenter une solution de Database Firewall ou un filtrage SQL au niveau du réseau pour bloquer les requêtes malveillantes.
Masquer les erreurs SQL détaillées côté utilisateur pour éviter la divulgation d'informations.
Activer les logs natifs d'AWS RDS pour la base de données et centraliser les traces dans un SIEM.
Restreindre l’accès réseau aux registres de packages, artefacts et modèles aux services internes autorisés uniquement (ex: registres d'images Docker, dépôts de code).
Implémenter un IAM centralisé pour l’accès aux dépôts de code, registries et pipelines CI/CD des microservices.
Appliquer le principe du moindre privilège aux comptes accédant aux registres d’artefacts et aux ressources ML (comme la base vectorielle pgvector).
Vérifier l’intégrité cryptographique des artefacts (documents, embeddings, scripts RAG) avant déploiement ou intégration.
Maintenir un AI Bill of Materials (AI-BOM) pour les modèles, datasets, dépendances et adaptateurs utilisés dans la chaîne RAG.
Scanner les images de conteneurs utilisées dans les pipelines ML et les microservices pour détecter les vulnérabilités.
Appliquer un rate limiting strict par compte, IP, session et endpoint d’authentification (LDAP, API).
Déclencher un account lockout temporaire ou un step-up MFA après N échecs d'authentification.
Imposer un MFA robuste (phishing-resistant si possible) sur les flux de login et de réinitialisation de mot de passe.
Détecter et bloquer les campagnes de credential stuffing à l’aide d’une analyse comportementale.
Générer tous les tokens et session IDs avec un CSPRNG (Cryptographically Secure Pseudo-Random Number Generator) et exiger une entropie suffisante.
Vérifier systématiquement la signature cryptographique de tout token d'authentification (ex: JWT) et valider les claims critiques côté serveur.
Protéger, faire tourner et cloisonner les clés de signature dans un stockage sécurisé (ex: AWS KMS, Secrets Manager).
Vérifier côté serveur que l'identifiant de ressource appartient à l'utilisateur du jeton JWT (ou de la session authentifiée), à chaque appel API.
Ne jamais extraire l'identité de l'appelant depuis la requête elle-même ; la lire uniquement depuis le jeton signé ou la session sécurisée.
Générer les identifiants de ressources en UUID v4 aléatoires pour empêcher l'énumération prévisible.
Utiliser des ACL (Access Control Lists) ou des règles d’accès fines lorsque les permissions doivent être gérées par utilisateur, groupe ou ressource.
Appliquer un contrôle d’accès par rôles (RBAC) ou attributs (ABAC) pour limiter chaque utilisateur aux actions et ressources autorisées selon son rôle.
Isoler les microservices par domaine de données : un service ne peut exposer que ses propres ressources ou celles pour lesquelles il est explicitement autorisé.
Journaliser et alerter sur tout accès à un identifiant n'appartenant pas au contexte de l'appelant.
Définir des DTOs (Data Transfer Objects) distincts par endpoint API, exposant uniquement les propriétés strictement nécessaires dans les réponses.
Éviter les sérialiseurs génériques ; lister explicitement les champs retournés dans chaque réponse.
Valider le payload entrant contre un schéma JSON strict (`additionalProperties: false`) pour interdire la soumission de propriétés non autorisées.
Restreindre la modification aux seules propriétés explicitement autorisées pour le rôle de l'appelant.
Mapper explicitement les champs autorisés vers le modèle métier et désactiver tout binding automatique non maîtrisé par Spring Boot.
Configurer l’API Gateway pour valider le payload entrant contre un schéma JSON strict.
Auditer les réponses API à chaque déploiement pour détecter les propriétés exposées par inadvertance.
Logger et alerter sur toute soumission ou exposition de propriété non autorisée comme événement de sécurité.
Implémenter un rate limiting strict sur les endpoints d’authentification (par IP, compte et device).

Implémenter un délai progressif (backoff exponentiel) entre les tentatives de connexion.
