Description de l'Application

Nom
assistant RH
Version
v1.0
Stack technique
FastAPI / Mistral / Gemini / pytm / WeasyPrint
L'assistant RH est une application web critique à usage interne, conçue pour automatiser les réponses aux questions des employés sur les congés, politiques internes et démarches administratives via un chatbot intégré. Architecturée en trois tiers (présentation, logique applicative, données) avec une approche microservices, elle s'appuie sur un frontend React communiquant en REST avec un backend Django, tandis que les données sensibles (données personnelles) sont stockées dans une base PostgreSQL relationnelle hébergée on-premise, partiellement dédiée par microservice sans communication directe entre eux. Le système intègre un modèle LLM Azure OpenAI en lecture seule, consommé via API externe sans fine-tuning ni RAG, et accessible uniquement via l'interface chatbot pour générer des réponses contextuelles sans déclencher d'actions externes. Aucune interaction directe entre microservices, pas de broker, de tâches asynchrones ou d'envoi d'emails, avec des logs non centralisés et une absence totale d'upload de fichiers. Les flux se limitent aux requêtes utilisateur (navigateur web), aux appels REST internes entre frontend et backend, et à la connexion sécurisée vers le LLM externe, tandis que la base de données reste isolée dans un réseau interne pour renforcer la protection des données sensibles.

Une BOLA survient lorsque l'API ne vérifie pas correctement si l'utilisateur authentifié a l'autorisation d'accéder ou de manipuler un objet spécifique qu'il demande (e.g., le dossier RH d'un autre employé) en se basant uniquement sur l'identifiant de l'objet dans la requête.

Mitigations

Mitigation à appliquer
Utiliser exclusivement des requêtes préparées avec des paramètres pour toutes les interactions avec la base de données PostgreSQL.
Interdire la concaténation dynamique SQL basée sur les entrées utilisateur dans le code du backend Django.
Valider strictement toutes les entrées utilisateur côté serveur (type, format, taille et contenu) avant de les utiliser dans les requêtes SQL.
Appliquer le principe du moindre privilège aux comptes d'accès à la base de données, en restreignant les privilèges SQL au strict nécessaire pour chaque microservice.
Mettre en place un mécanisme de masquage des données pour limiter l’exposition des données sensibles aux utilisateurs non autorisés.
Masquer les messages d'erreur SQL détaillés côté utilisateur pour éviter la divulgation d'informations sur la structure de la base de données.
Activer le chiffrement TLS1.3 sur les flux de communication entre le backend Django et la base de données PostgreSQL.
Journaliser tous les événements de sécurité applicatifs et de base de données, et centraliser les traces dans un SIEM pour détecter et alerter sur les anomalies et requêtes suspectes.
Implémenter un mécanisme d’échappement systématique de toutes les données en sortie, en fonction du contexte (HTML, JavaScript), avant leur affichage par le frontend React.
Valider et filtrer strictement toutes les entrées utilisateur côté serveur (backend Django) pour supprimer les caractères spéciaux et balises potentiellement malveilluses.
Implémenter une politique de sécurité de contenu (CSP) stricte pour restreindre l’exécution de scripts non autorisés et le chargement de ressources externes dans le navigateur des utilisateurs.
